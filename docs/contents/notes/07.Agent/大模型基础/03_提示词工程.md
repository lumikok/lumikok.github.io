# 提示词工程

## 一、基础概念

### 1. 两类核心 Prompt

1. **用户 Prompt**
    
    用户主动输入的对话内容，需求零散、格式随意、不可控，偏向「提需求」。
2. **系统 Prompt（开发核心）**
    
    开发人员提前写入代码、框架配置中的固定提示词，用户不可见、不可修改。
    
    作用：定义 AI 身份、行为规则、执行流程、输出规范，是 Agent 的底层运行准则。

### 2. 提示词工程定义

不修改大模型权重、不做微调，通过**设计、优化、约束提示词**，规范模型决策逻辑、工具调用、输出格式，让 AI 稳定完成指定业务任务的工程化手段。

## 二、Agent 开发中提示词工程的核心作用

1. **约束行为，杜绝幻觉**
    
    强制模型不编造数据、不虚构业务内容，必须依赖数据库、接口、工具获取真实信息。
2. **规范工具调用**
    
    明确何时调用工具、可用工具范围、入参格式、调用频次，避免乱调用、重复调用、参数错误。
3. **统一结构化输出**
    
    限制返回格式为 JSON、固定列表、标准字段，适配后端解析、前端渲染、程序自动化处理。
4. **标准化决策逻辑**
    
    通过思维链约束，让模型分步拆解任务、逻辑推导，适配复杂多步骤业务。
5. **实现多智能体分工**
    
    为不同 Agent 划定职责边界、沟通格式、工作范围，防止功能冲突、重复作业。
6. **降低开发维护成本**
    
    模型行为异常优先修改提示词，无需改动大量业务代码，迭代效率更高。

## 三、通用 Prompt 万能结构（开发直接复用）

### 1. 四大固定模块

- 角色定位：明确 Agent 身份、业务场景、核心职责
- 执行规则：强制要求、禁止行为、数据使用规范
- 任务流程：任务拆解步骤、处理顺序、多轮交互规则
- 输出约束：格式要求、内容限制、冗余内容剔除规则

### 2. 通用模板

plaintext

```
你是【指定业务】专属智能体，专注完成核心工作。
严格遵守以下规则：
1. 所有客观数据必须通过工具/数据库查询，禁止编造、虚构内容；
2. 处理需求前，先拆解问题、分析核心诉求，分步完成任务；
3. 严格按照指定工具列表执行操作，禁止私自调用未授权工具；
4. 最终输出严格遵循固定格式，无多余废话、无无关闲聊；
5. 遇到超出能力/无数据场景，统一固定话术回复，不随意拓展。
按以上要求，处理当前用户请求。
```

## 四、Agent 开发必学核心技术

### 1. 思维链 CoT

核心：强制模型**分步思考、逐步推理**。

用途：复杂业务拆解、逻辑计算、多条件筛选类任务，大幅提升准确率。

用法：在系统提示中加入要求：`请先分步分析问题，梳理关键信息后，再给出最终答案`。

### 2. 结构化输出约束

核心：强制模型输出可被代码解析的标准化内容。

常用约束：固定 JSON 格式、限定 key 字段、禁止 markdown、纯文本返回。

### 3. 少样本示例（Few-Shot）

核心：给出 1~3 条输入输出样例，让模型模仿格式与逻辑。

用途：格式要求严格、业务规则特殊的定制化场景。

### 4. 工具调用专属约束

- 区分「可直接回答」和「必须调用工具」的场景
- 限制单次仅调用一个工具，等待返回结果后再下一步操作
- 规范工具参数书写规则，避免语法错误

### 5. 多智能体协作提示

- 各 Agent 独立系统提示，职责隔离
- 统一跨 Agent 消息交互格式
- 划定读写权限、数据传递范围

## 五、开发常见误区

1. 误区：有 Agent 框架就不需要提示词
    
    纠正：LangChain、LlamaIndex 仅提供调度能力，**决策逻辑、行为约束完全依赖提示词**。
2. 误区：提示词越长效果越好
    
    纠正：只保留核心业务规则，冗余内容会干扰模型判断，增加 token 消耗。
3. 误区：忽略禁止性规则
    
    纠正：只写要做什么，不写不能做什么，极易出现幻觉、越权、闲聊跑偏。
4. 误区：多 Agent 共用一套提示词
    
    纠正：无职责区分会导致功能混乱、任务重叠，多智能体必须差异化配置。

## 六、核心总结

1. 应用层 Agent 开发**无需深度学习花式文案提示**，只聚焦工程化约束能力；
2. 提示词工程是 Agent 的「规则手册」，代码决定能力上限，提示词决定运行下限；
3. 所有落地级 AI 项目，系统提示词都是核心配置，是保障项目稳定运行的关键；
4. 学习重心：规则约束、工具管控、结构化输出、多智能体分工，贴合项目落地需求。
