# 大模型调用



## 网络基础知识

### IP地址

1. **定义**：IP地址是**联网设备在互联网中的唯一标识（网络身份证）** 用于在网络中定位设备，实现数据传输与设备间通信。

2. **地址位数**：
	- **IPv4**：32 位二进制
	- **IPv6**：128位二进制
3. **公网与内网**：
	- **公网IP**：互联网全局唯一，全网所有设备均可直接访问
	- **内网IP**：仅在局域网内有效，外网无法直接访问
4. **双栈网络**：
		目前设备会同时拥有 `IPv4` 和 `IPv6` 两个地址，两种协议并行兼容

**IPv4地址样例如下**:

![](1.jpg)

- 上面一行：日常使用、代码里写的「点分十进制 IP」
- 下面一行：电脑 / 路由器硬件底层原生识别的「二进制 IP」
- 32位二进制，平均切成4份，每一份固定8位二进制

**注意**：
- `127.0.0.1 是一个非常特殊的IP地址，表示的是本机地址（也称为本机回环地址）
- **核心用途**：
	- **本地服务测试**：验证本地启动的 Web 服务、API 接口是否正常。 访问路径：`http://127.0.0.1:端口号`
	- **网络连通性测试**：`ping 127.0.0.1，检测本机网络协议是否正常
- **重要规则**：
	- 仅限本机访问，外部设备无法通过 127.0.0.1) 访问你的服务
	- 等价于 `localhost`（主机名）
	- 数据仅在本机内部流转，不经过外网网卡，速度最快

### 域名

### HTTP协议

#### 核心架构概览

DeepSeek 大模型调用采用经典的 **客户端 - 服务端（C/S）架构**，基于 HTTP/HTTPS 协议完成请求 - 响应交互。

```plaintext
客户端（发起请求） ←→ 网络（IP/域名+端口） ←→ 服务端（DeepSeek 模型推理）
```

#### 架构各角色解析

##### 1. 客户端

- **定义**：发起请求的程序，如 Python 脚本、浏览器、APP、命令行工具等。
- **核心职责**：
    
    - 构造请求参数（prompt、模型名、生成参数等）
    - 携带鉴权信息（API Key）
    - 发送 HTTP 请求并解析响应结果
    - 维护对话上下文（HTTP 无状态，服务端不保存历史）
    
	[具体代码示例](#代码调用大模型)； ^ab0bc4
##### 2. 网络层（通信核心）

- **关键要素**：
    
    - **域名 / IP**：DeepSeek API 域名 `api.deepseek.com`
    - **端口**：HTTPS 默认端口 `443`
    
- **协议要求**：必须使用 **HTTPS**，通过 TLS 加密防止 API Key 泄露与请求篡改。

##### 3. 服务端（DeepSeek）

- **核心职责**：
    
    1. 鉴权验证：校验 API Key 合法性、权限与调用额度
    2. 请求解析：解析 JSON 格式的请求体
    3. 模型推理：调用指定模型（如 `deepseek-chat`）生成响应
    4. 结果返回：封装 JSON 格式的响应数据返回客户端

#### 网络模型

为了统一程序在网络中通信的模型和标准，构建了以下模型：
- **OSI网络模型**：全球网络互连标准模型
- **TCP/IP网络模型**：可以认为是OSI的简化版

![697](1%201.jpg)

##### TCP/IP 各层详解

- **应用层**：将用户与应用程序交互的数据按照协议格式进行封装
- **传输层**：负责将数据准确送到对应的应用程序（端口）
- **网络层**：负责基于IP地址将数据包路由给对应设备
- **网络接口层**：负责数据在物理网络中的传输、处理与硬件设备交互

#### HTTP协议介绍

- **概念**：`Hyper Text Transfer Protocol`，超文本传输协议，规定了客户端和服务器之间数据传输的规则（只有在请求和响应中都遵循了统一的规则，服务端才能读懂客户端发送来的请求，客户端才能解析服务端响应的结果）
- **特点**：
	1. **基于文本的协议**：请求和响应的部分的协议内容为文本格式，底层通过TCP协议传输，稳定性强
	2. **基于请求 - 响应模型**：一次请求对应一次响应，必须由客户端先发起请求，服务端才会返回响应
	3. **无状态**：服务端不会记忆与客户端的历史交互信息，每次请求 - 响应都是独立的

####  HTTP协议 - 请求数据格式

##### 整体结构

一条标准的 `HTTP` 请求由三部分组成：

1. 请求行
2. 请求头（Headers）
3. 空行 + 请求体（Body）

##### 各部分说明

###### 1. 请求行

- **格式**：请求方法 接口路径 协议版本（一般统一用 **`POST`**）
- **实例**：
```http
POST /chat/completions HTTP/1.1
```

**请求方法**：
- `GET`：请求参数在请求行中，没有请求体，该方式的请求参数大小在浏览器中是有限制的
- `POST`：请求参数在请求体中，`POST`请求大小是没有限制的
###### 2. 请求头

以 `键: 值` 形式，配置权限、数据格式等。

**开发必记 2 个**：

- `Content-Type: application/json`：请求体是 JSON 格式
- `Authorization: Bearer 密钥`：接口鉴权，大模型调用必备

###### 3. 请求体

- **GET 无请求体**
- **POST 才有请求体**
- 大模型所有提示词、模型参数、对话上下文，全部放在 JSON 请求体中

##### 完整极简示例

```http
POST /chat/completions HTTP/1.1
Host: api.deepseek.com
Content-Type: application/json
Authorization: Bearer sk-xxx 

{"model":"deepseek-chat","messages":[{"role":"user","content":"你好"}]}
```

####  HTTP协议 - 响应数据格式

##### 整体结构

与请求对应，`HTTP` 响应分为 3 部分：

1. 响应行（状态行）
2. 响应头
3. 空行 + 响应体

##### 各部分详解

###### 1. 响应行

格式：`协议版本 状态码 状态描述`

示例：
```http
HTTP/1.1 200 OK
```

常见核心状态码（开发必记）：

- `200`：请求成功
- `401`：鉴权失败（密钥错误）
- `429`：请求限流 / 次数超限
- `400`：请求参数错误
- `404`：请求资源不存在，url输入有误，或者网站资源被删除了
- `500`：服务端内部错误

###### 2. 响应头

键值对格式，由服务端返回

一般**开发不用手动配置**，只需要偶尔查看排查问题。

###### 3. 响应体

- 核心数据存放位置
- 大模型、API 接口**全部返回 JSON 格式**
- 最终拿到的回答、token 消耗、模型信息，都在这里

## 接口测试工具：Apifox

### Apifox 介绍

`Apifox` 是一款`API`设计、开发、测试的一体化协作平台，是项目开发中进行`API`接口测试的工具

### 何为API

- **介绍**：**`API` = 应用程序编程接口**，全称：`Application Programming Interface`；
- **核心作用**：让**两个不同的软件 / 程序**互相沟通、交换数据、调用功能
	- **举例**：
		1. 你用 AI 聊天（DeepSeek）→ 你的软件调用**大模型 API**
		2. 天气 APP 显示实时天气   → 调用气象平台的**天气 API**
		3. 网站微信支付 → 调用**微信支付 API**
- **本质**：**固定格式的网址 + 参数 + 规则**，发一段数据过去，对方按规则返回结果

### JSON格式

`JSON(JavaScript Object Notation)`是前端的一种对象表示方法。表示形式类似于`Python`中的字典，都是键值对这种形式，不过所有的`key` 都必须使用双引号引起来，值可以是任何类型：

- **对象**：用`{}`表示，`{}`之间是键值对形式，键是字符串，值可以是任意其它类型
- **数字**：整数和小数
- **字符串**：用`""` 引起来
- **布尔**：`true`和`false`
- **列表**：用`[]`表示，`[]`中是列表的元素，多个元素以`,`分割

完整调用示例如下：
```JSON
curl https://api.deepseek.com/chat/completions
	-H "Content-Type: application/json"
	-H "Authorization: Bearer ${DEEPSEEK_API_KEY}"
	-d '{  
	  "model": "deepseek-v4-pro",  
	  "messages": [  
		{"role": "system", "content": "You are a helpful assistant."},  
		{"role": "user", "content": "Hello!"}  
	  ],  
	  "thinking": {"type": "enabled"},  
	  "reasoning_effort": "high",  
	  "stream": false  
		}'
```

**解释**：

- **curl**：命令行发送 HTTP 请求的工具
- **[https://api.deepseek.com/chat/completions](https://api.deepseek.com/chat/completions)**：

    大模型接口地址（服务端地址）
- **-H**：表示 HTTP 请求头
- **Content-Type: application/json**：

    告诉服务端：我发的数据是 **JSON 格式**
- **Authorization**：鉴权（身份验证）
- **Bearer**：固定格式
- **${DEEPSEEK_API_KEY}**：你的 API Key
- **-d**：HTTP 请求体（POST 数据）
- - **model**
    
    使用的模型：`deepseek-v4-pro`
    
- **messages** 对话上下文
    
    - `system`：给 AI 设定角色（你是一个有用的助手）
    - `user`：用户提出的问题（Hello!）
    
- **thinking: enabled**
    
    开启**深度思考**
    
- **reasoning_effort: high**
    
    思考强度：高
    
- **stream: false**
    
    关闭流式输出，**一次性返回全部结果**

## 会话记忆

上述接口测试过程中，与`AI`大模型的交互本质是无状态的，每一次请求响应都是相互独立的，**即无上下文记忆能力**

### 解决方案

#### 1. 原生会话记忆（默认模式）

##### 原理

大模型 API 无服务端记忆，**每轮请求手动拼接完整历史对话**，连同新问题一起传入 `messages`。

##### 致命问题

对话越多上下文越长，**Token 滚雪球式无限累积**：

- 费用变高、响应变慢
- 触发上下文窗口上限
- 冗余信息干扰回答质量
#### 2. 四种工程解决方案

##### 方案一：滑动窗口截断（最简轻量）

只保留最近 N 轮聊天，早期记录直接删除。

- 优点：代码简单、零额外开销
- 缺点：丢失久远记忆
- 适用：短对话、简单问答、小型 Demo

##### 方案二：对话摘要压缩（性价比最高）

每隔若干轮，让大模型**总结精简全部历史**，用一段短摘要替换完整聊天记录。

- 优点：保留关键信息，大幅缩减上下文
- 缺点：多一次模型调用消耗
- 适用：多轮聊天、普通 AI 助手、Agent 基础版

##### 方案三：向量库 + RAG 长效记忆（高阶方案）

1. 历史对话向量化存入向量数据库
2. 新问题进来，语义检索**只拉取相关历史片段**
3. 无关记录不进上下文，彻底杜绝无限累加

- 优点：永久长期记忆、精准关联上下文
- 缺点：架构复杂，需嵌入模型 + 向量库
- 适用：多智能体、个性化系统、长期业务项目

##### 方案四：厂商托管 Session（过渡方案）

依靠平台 `session_id` 云端存聊天记录。

- 本质：只是厂商帮你拼接历史，**依旧存在滚雪球问题**
- 适用：快速原型、临时测试，不适合正式项目

#### 3. 选型总结

- 临时 Demo → 原生拼接 + 滑动窗口
- 常规多轮项目 → 滑动窗口 + 摘要压缩
- 复杂 Agent / 长期个性化业务 → RAG 向量记忆

### 示例代码（DeepSeek）

[具体底层原理](#^ab0bc4)；

```python
# Please install OpenAI SDK first: `pip install openai`
import os
from openai import OpenAI

# 创建与AI大模型交互的客户端对象
client = OpenAI(
	# 读取的系统环境变量（方法一）
    api_key=os.environ.get('DEEPSEEK_API_KEY'),
    # 方法二，直接在此处写自己的APIKEY
    # api_key = ("APIKEY")
    base_url="https://api.deepseek.com")

response = client.chat.completions.create(
    model="deepseek-v4-pro", # 指明调用的是哪个模型  - 即模型ID
    messages=[
        {"role": "system", "content": "You are a helpful assistant"},
        {"role": "user", "content": "Hello"},
    ],
    stream=False,
    reasoning_effort="high",
    extra_body={"thinking": {"type": "enabled"}}
)

print(response.choices[0].message.content)
```

### 原理

#### 简单阐述

- **总流程简介**： Python 代码 → 打包成一段 JSON → 发给大模型服务器 → 服务器计算 → 返回一段 JSON → 你解析取出答案
- **本质**：**一次标准的网络请求（POST )**，只不过是使用的`OpenAI`打包好的请求工具

#### 创建 client 客户端

```python
client = OpenAI( 
	api_key=os.environ.get('DEEPSEEK_API_KEY'),
	base_url="https://api.deepseek.com" 
)
```

它只是**存了两个信息**：

1. **api_key** = 你的身份证（验证你是谁、有没有钱、有没有权限）
2. **base_url** = 目标地址：[https://api.deepseek.com](https://api.deepseek.com)

client 本身不发请求！它只是一个 **“准备好发请求的工具”**。

#### 核心：真正发请求的地方

```python
response = client.chat.completions.create(
	 model="deepseek-v4-pro", 
	 messages=[...], 
	 stream=False, 
	 ... 
)
```


 **它做了这 4 步：**

1.  把你写的所有参数，打包成一段 JSON
2.  向 `base_url` 发送 **POST 请求**
3.  在请求头里带上你的 api_key
4.  等待服务器计算、思考、返回结果

#### 服务器返回的结果

返回一段**JSON**，长这样：

```json
{ 
	"choices": [ 
	{ 
		"message": { 
		"content": "Hello! How can I assist you today?" 
		} 
	} 
  ] 
}
```

因此取返回的聊天内容必须写成：

```python
response.choices[0].message.content
```
